### 容器职责
| 容器 | 镜像 | 职责 | 调度 |
|------|------|------|------|
| `radar` | `python:3.11-slim` | 全市场扫描横盘暗筹信号→落库；周期采集价格快照；Telegram 推送 | 内部循环：每小时 1 次 oi 扫描 + 价格快照 |
| `api` | `python:3.11-slim` + FastAPI + uvicorn | 查询信号详情、实时算 PnL、策略聚合统计 | 常驻，端口 8000 |

### 数据流
1. `radar` 每小时跑 `oi` 模式 → 三策略评分结果**逐条落 `signals` 表**
2. `radar` 每小时跑一次**价格快照采集**：对所有活跃信号 symbol 拉最新价，落 `price_snapshots`
3. `radar` 落完信号调用 `send_telegram`（保留原有推送）
4. `api` 收到请求时**实时计算 PnL**：`entry_price` 取自 `signals`，`current_price` 实时拉币安或取最新快照，历史指标（max_high/pnl_Nd）从 `price_snapshots` 查算

### 共享存储
- SQLite 文件 `accumulation.db` 挂载到两容器共享卷（`./data/` 目录）
- 两容器读写同一文件：radar 写，api 读（SQLite WAL 模式支持并发读+单写）
- 启用 `PRAGMA journal_mode=WAL` 提升并发读

## 3. 雷达改造点（`accumulation_radar.py`）

### 3.0 信号重构（核心）
旧 5 合约策略（chase/combined/ambush/hot/highlight）全部删除。当前生产策略为两个：

**策略 A `sideways_acc` 横盘暗筹**（保留，回测 95 天 154 信号、超额 +0.79%、胜率 48.7%，跑赢基准）。

**策略 B `fingerprint` 庄家拉盘指纹库匹配**（新增，90 天样本外 289 命中、期望 +3.11%、胜率 52.9%、肥尾捕获 5.2%、跌>10%仅 8%，本项目首个通过样本外验证的正期望信号）。

#### 3.0.1 指纹库策略原理
基于 ERA/ESPORTS/RE/BANK 四币庄家行为实证：不追求一套参数适配所有庄家，而是**采集真实拉盘事件起飞前14天的量价“编舞”指纹**，下次见到相似形状即预警。

- 指纹 = 归一化价格形状(14日 min-max) + 量比轨迹(14日/中位) + 行为签名(试盘脉冲/洗盘/波动率收敛/止跌企稳/探底/低位)。
- 相似度 = 0.4×价格形状相关 + 0.3×量比形状相关 + 0.3×签名匹配率。
- **关键发现**：纯形状相似是噪音(样本外-0.49%)，因为下跌币和起飞前形状太像；**形状相似 + 正确行为签名**才产生正期望。分化器是「波动率收敛 + 止跌企稳 + 低位」三签名。
- 庄家策略本质是低胜率高 payoff 肥尾策略，**评判用期望值/肥尾捕获率/风险收益比RR，不以胜率为唯一标准**。

#### 3.0.2 指纹库生产配置（锁定）
```
strategy = fingerprint
similarity >= 0.65  AND  波动率收敛(converge=True)  AND  止跌企稳(standstill=True)  AND  低位(pos<0.3)
```
- 库文件：`data/fingerprints.json`（19 条指纹，A=1/B=3/D=15，来自90天83个拉盘事件）。
- 调度：每 6 小时扫一次全市场。同币命中后冷却5日，取最高相似度。

#### 3.0.3 指纹信号① 横盘暗筹触发条件（保留）

**信号① 横盘暗筹触发条件**（t 时刻，仅用 ≤t 数据）：
- 现货 5 日振幅 < 8%（横着不涨）
- 现货 5 日涨幅 < 3%（未拉升）
- 现货量比 = 近24h均量 / 前7日均量 > 1.1（量温和放大）
- 合约资金费 <= 0.01%（中性偏负，散户看空、庄家未拉合约）
- 5 日涨幅 > -8%（排除已下跌趋势）
- **实盘叠加加分**（非必要）：OI 5日斜率 > 0（OI 暗流上行）→ score 加分

**扫描对象**：全市场 530 永续（不再依赖 pool 池；横盘暗筹要在全市场找，不在已异动的池里找）。
**symbol 映射**：合约→现货（1000X→X、同名、无现货则跳过），缓存到 `pool_state` 或内存。

### 3.1 配置
- 保留 `DB_PATH = ./data/accumulation.db`（默认值，可被环境变量覆盖）
- 保留 `TG_BOT_TOKEN` / `TG_CHAT_ID`（从 `.env.oi` 读，已配置 token，待填 chat_id）
- 启动时 `PRAGMA journal_mode=WAL; PRAGMA synchronous=NORMAL;`

### 3.2 数据落地
- `init_db()`：建上述 3 张表 + 索引（替换原 watchlist/alerts 表，保留兼容迁移）
- `save_watchlist()`：改写 `pool_state`（每次 pool 跑完写一条快照）

def collect_price_snapshots(conn):

### 4.3 PnL 实时计算（`/signals/{id}/pnl`）
```python
def compute_pnl(signal, conn, live=True):
    entry = signal.entry_price
    # 最新价：实时拉币安，失败则用最新快照
    if live:
        current = fetch_live_price(signal.symbol) or get_latest_snapshot_price(...)
    else:
        current = get_latest_snapshot_price(...)
    # 历史指标：从 price_snapshots 查
    snaps = get_snapshots_since(signal.symbol, signal.signal_time)
    max_price = max(s.price for s in snaps) if snaps else None
    min_price = min(s.price for s in snaps) if snaps else None
    pnl_1d = find_snapshot_at(signal, +1*24) ...  # 1天/7天/30天那个小时快照
    return {
        "entry_price": entry,
        "current_price": current,
        "unrealized_pnl_pct": (current - entry) / entry * 100,
        "pnl_1d_pct": ..., "pnl_7d_pct": ..., "pnl_30d_pct": ...,
        "max_high_pct": (max_price - entry) / entry * 100 if max_price else None,
        "max_low_pct": (min_price - entry) / entry * 100 if min_price else None,
        "curve": [{"ts": s.ts, "price": s.price, "pnl_pct": (s.price-entry)/entry*100} for s in snaps],
    }
```

## 5. PnL 计算逻辑

### 5.1 定义
- **买入点**：信号 `signal_time` 时刻的 `entry_price`
- **持有方式**：买入持有（Buy & Hold）
- **未实现盈亏**：`(current_price - entry_price) / entry_price × 100%`
- **跟踪期**：默认 5 天；超期信号不再采价，但历史 PnL 曲线保留、仍可查询
- **实时性**：`current_price` 默认实时拉币安 `/fapi/v1/ticker/price`（轻量单币接口），失败回退最新快照

### 5.2 关键指标
| 指标 | 计算 |
|------|------|
| `unrealized_pnl_pct` | 实时：`(current - entry)/entry` |
| `pnl_Nd_pct` | 用 N 天后那个小时快照的 price 算（无快照返回 null） |
| `max_high_pct` | `(max_price - entry)/entry`，跟踪期内快照最高 |
| `max_low_pct` | `(min_price - entry)/entry`，跟踪期内快照最低 |
| `win_rate` | `unrealized_pnl_pct > 0` 的信号占比 |
| `hit_rate_X` | `max_high_pct >= X` 的占比 |

### 5.3 平仓策略（API 可选参数 `exit=`）
- `hold`（默认）：持有到现在，用 `current_price`
- `fixed_Nd`：持有 N 天平仓，用第 N 天快照价算 realized PnL
- `tp_sl`：止盈 X% / 止损 Y%，用快照序列模拟触发时刻

### 5.4 数据精度
- 价格快照频率：每小时 1 次
- PnL 曲线点数：5 天 ≈ 120 点/信号
- 下架处理：连续 24 小时无快照 → 标 `delisted`，PnL 按最后一次价计算并标记

## 7. 决策记录（已确认）

| # | 决策项 | 选择 | 理由 |
|---|--------|------|------|
| 1 | API 技术栈 | **FastAPI + uvicorn** | 现代、自带 `/docs` 可当简易看板、性能好 |
| 2 | PnL 平仓策略 | **只做 Buy & Hold**（默认） | 跟踪期仅5天，tp_sl 回放复杂度不值；`exit` 参数留接口 |
| 3 | 跟踪期 | **5 天** | 用户确认，只关心短期 |
| 4 | 信号落库范围 | **每策略 Top 10** | 折中：质量与样本量平衡 |
| 5 | SQLite 并发 | **WAL 模式共享卷** | 写入量小（每小时百级行），够用 |
| 6 | Telegram | **保留**，token+chat_id 已配 `.env.oi` | 用户确认 |
| 7 | 前端 | **只要 API** | FastAPI `/docs` 充当看板，后续可加 |
| 8 | PnL 公式方向 | **A：(current-entry)/entry** | 常规，涨=正盈亏 |
| 9 | 网络 | 全通，无需代理；Coinglass 不引入 | 实测 fapi/www.binance/coingecko 均 200 |
