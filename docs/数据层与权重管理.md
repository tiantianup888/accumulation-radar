# 数据层与 API 权重管理

## 概述
市场原始数据（量价日K / funding / OI）从币安合约 API 拉取，统一存入 DuckDB
（`data/market.duckdb`）。信号/快照等运营数据仍留在 SQLite（`data/accumulation.db`）。
DuckDB 列存+压缩，适合分析型查询；SQLite 保留事务型写入。

## API 权重管理设计（核心：binance_client.py）

### 权威限制来源
- `exchangeInfo.rateLimits`：**REQUEST_WEIGHT 2400/分钟**（唯一文档化的限制）。
- 端点权重（实测响应头 `x-mbx-used-weight-1m` 累计）：
  - klines：limit 100→权重1, 500→2, 1000→5, 1500→10
  - exchangeInfo/premiumIndex/time：权重1
- **fundingRate 与 `/futures/data/*`（openInterestHist 等）不返回权重头**，走独立的 WAF
  突发保护，返回 **403 Forbidden**（非 429）。这是未文档化的第二套限制。

### 两套限制，两种对策
1. **REQUEST_WEIGHT 2400/min（→429/418）**：
   - 跟踪响应头 `x-mbx-used-weight-1m`；接近预算 1800 时睡到下一分钟窗口。
   - 429/418：按 `Retry-After` 指数退避。
2. **WAF 突发保护（→403）**：
   - 实测：fundingRate 持续 ~100-125 次/2min 即触发 IP 级 403（**并发会立即触发**，曾因 12 线程 burst 被封）。
   - ban 是 transient（停手 1-2 min 自清），**但封禁期内重试会续期**。
   - 对策：① 全局串行（绝不并发）；② data 端点 1s/请求间隔；③ **主动节流**：每 80 次 data 请求休 30s，避开 ban 阈值；④ 403 冷却 base 120s 指数到 1800s，给足自清时间，不快速重试。

### 实测效果
| 拉取 | 币数 | 403次数 | 429次数 | 结果 |
|---|---|---|---|---|
| funding（无主动节流） | 530 | 8 | 0 | 自动恢复，0 失败 |
| klines（标准桶） | 530 | 0 | 0 | 权重节流，0 失败 |
| OI（主动节流） | 530 | **0** | 0 | **主动节流完全避免 WAF ban** |

→ 主动节流把 OI 的 8 次 ban 降到 0，验证设计有效。

## 模块
- `binance_client.py`：权重管理客户端（weight 追踪 + WAF 主动节流 + 403/429 退避）。
- `duckdb_store.py`：DuckDB 存储（staging 表 + 批量 DELETE+INSERT 幂等写入，
  比 INSERT OR REPLACE executemany 快 140×）。
- `fetch_funding.py`：funding 拉取（`--days`，分页）。
- `fetch_market_data.py`：日K + OI 拉取（`--kind klines|oi|all`）。

## 复用
- 权重预算 1800（留 25% 余量）；data 端点 1s 间隔 + 每 80 次休 30s。
- 可调整 `BinanceFuturesClient(std_delay=, data_delay=, weight_budget=)`。
